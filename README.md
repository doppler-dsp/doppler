# doppler

A multi-module DSP library built with
[just-makeit](https://github.com/just-buildit/just-makeit).

Each module is a clean, jm-native implementation — C-first, no wrapper layers.

---

## The rule: re-implement, do not wrap

Every algorithm here is written from scratch using just-makeit's scaffolding
pattern.  Doppler's C source is the **reference**, not the dependency.

**Do not do this:**

```c
/* WRONG — wraps the existing dp_ type, adds nothing */
float acc_f32_step(acc_f32_state_t *state, float x)
{
    return dp_acc_f32_step(&state->dp, x);
}
```

**Do this:**

```c
/* RIGHT — jm-native state, jm-native step, same algorithm */
static inline void
acc_f32_step(acc_f32_state_t *state, float x)
{
    state->acc += x;
}
```

The distinction matters because:

- `step()`, `steps()`, and `reset()` are generated by jm around your
  implementation — wrapping a foreign type breaks that contract
- jm generates the Python glue, benchmarks, and type stubs from the
  scaffold; a wrapper prevents jm from doing its job
- The state struct must be owned by jm so `create`/`destroy`/`reset`
  work correctly and the Python lifetime model holds

---

## The default: step / steps

> ## `step()` / `steps()` is the default. Deviate only with a very compelling reason.
>
> **If your algorithm can be expressed as "consume one sample and optionally
> produce one output", use `step()`.  `steps()` is generated for free.
> Named methods (`dump`, `madd`, `flush`, …) are for operations that
> genuinely cannot be expressed as a single-sample step — not for convenience.**

The accumulator is the canonical example of the pattern done right:

- `step(x) → void` with `--mutable` — push one sample, no output
- `steps(x[])` — auto-generated batch loop, zero extra code
- `dump()` as a named method — because atomic read-then-zero cannot be a step variant

If you are reaching for a named method when a plain `step()` would work,
stop and reconsider.

## Reference example: accumulator

The cleanest illustration of this pattern lives in the just-makeit repo:

```
just-makeit/src/just_makeit/examples/accumulator/
```

Read `README.md` there end to end before writing any module here.  It shows:

- How `step(x) → void` with `--mutable` becomes a push operation
- How `steps(x[])` (auto-generated) is batch-add — no code required
- Why `dump()` must be a named method (atomic read-then-zero, can't be expressed as step)
- The `madd(x[], h[])` pattern for weighted accumulation
- How `just-makeit perf` + explicit `JM_RESTRICT` + `JM_VEC_F32` delivers
  an 11× throughput gain over the scalar baseline (1.66 → 18.11 G samples/sec)

Run it end-to-end in one command:

```bash
just-makeit example accumulator
```

| `native/inc/acc_cf64/acc_cf64_core.h` | `jm object` | Yes | Same fix for `double complex[] x`; implemented `step()` inline |
| `native/src/acc_f32/acc_f32_core.c` | `jm object` | Yes | Implemented `get`, `dump`, `madd`, `add2d`, `madd2d` (bodies were `/* <<IMPLEMENT>> */` stubs) |
| `native/src/acc_cf64/acc_cf64_core.c` | `jm object` | Yes | Same; cf64 weights cast to `double` before multiply |
| `native/src/accumulator/accumulator_ext.c` | `jm object` | No | Python C extension glue — entirely generated |
| `native/src/accumulator/CMakeLists.txt` | `jm object` | No | |
| `native/src/accumulator/accumulator_core.c` | `jm object` | No | Module-level C (lifecycle only for a sink) |
| `native/inc/accumulator/accumulator_core.h` | `jm object` | No | Module umbrella header |
| `native/tests/test_acc_f32_core.c` | `jm object` | No | C smoke test |

>
> **Run the scaffold. Implement the two `.c` bodies. Fix the test file for your return types. Leave everything else untouched.**
